Official Nxt Releases / Re: NRS v1.2.6
« on: August 21, 2014, 07:10:18 pm »
Quote from: Eadeqa on August 21, 2014, 07:04:47 pm
Quote from: Jean-Luc on August 21, 2014, 06:52:25 pm
It is an API because it defines another way to submit transaction parameters. It is a kludge which may seem to work but will create maintenance problems later, and when I finally decide to get rid of it, it will turn out somebody has become reliant on it.

It's regular transaction API. There is no new API. The nxt server just parses the address to check if a message is included with the address. Otherwise, it's regular address. 

It's not a "kludge" as it can be very useful.

For example, If I want to sell something, instead of generating a new address for each order, I can just do this:

NXT-GZYP-FMRT-FQ9K-3YQGS/messageEncrypted=53
NXT-GZYP-FMRT-FQ9K-3YQGS/messageEncrypted=52

where 52 and 53 are order numbers. That way I can track who made the payment without generating new addresses for each order.
whatever you call it, API or "chickens on a rooftop", it is code in the NXT core that has to stay there, or we lose backward compatibility with any place that relies on "chickens on a rooftop"

so unless this is something to support long term, with a 6 month phaseout if we ever decide we want to not be backward compatible, then it should not be added in the first place. Now, that being said, some mechanism to allow reuse of addresses for multiple tx, and maybe this is not so bad a way to do that

James

P.S. I like to call this API change than "chickens on a rooftop"
