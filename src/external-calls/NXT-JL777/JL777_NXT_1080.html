I have pushed new versions to the dev MGW servers and the bridge for mgwBTCD. I have tested realtime deposit address generation, deposit recognition, deposit triggering an asset transfer (and publishing pubkey), withdraws being detected and also withdraws being processed when all three servers are in total agreement. This is only accessible via the jnxt.org URL for now and deposit/withdraw processing needs to be manually done. I put in a "softwall" so that only nodes on the whitelist are able to even get the SuperNET packets through to the code. It is also in an isolated private supernet. The bridge is currently technically not a bridge as it is just on the same private supernet and it is bridging via the PHP web interface.

you can get status of the servers by: http://jnxt.org/init/?requestType=status&pubkey=734b83479469164e6059b98c1679043a278c1ba8d18d1d42d348d255baf2f656&NXT=NXT-MEXA-RJSP-NKDU-FWWHM&coin=BTCD
basically any NXT/pubkey that isnt used will do

now if you make it  a "newbie" call, it will create a multisig address and you can then send a deposit to that address. if "convertNXT" field is set (eventually) you will get in your account a mix of the mgwCOIN asset + NXT. this is not done yet but it is the link between MGW and InstantDEX, so it will be done next week. you can also specify "email" field and it will email you the JSON. not very friendly, but this is for debugging (and also I think the ssmtp on the bridge is not quite ready yet)

what is missing is to automatically process deposits and withdraws when the servers are in consensus. Also, large status reports dont fit and I havent had a chance to get it properly processed on the receiving bridge node. So, to see the actual results for now you can get static HTML: http://209.126.70.159/MGW/msig/ALL or http://209.126.70.159/MGW/msig/<NXTnumericaladdr>

the above msig dir is for multisig addrs and the following are for the status returns:  http://209.126.70.159/MGW/status/ALL or http://209.126.70.159/MGW/status/<NXTnumericaladdr>

Hi jl777,

Thanks, I will have another look at JLH thread.

At the time, I didn't understand very well the proposed solution; from what I undestood, the answer bits had different weights and you would compute first the bits with highest weight, then at the end you did not have a big influence over the end result, something like computing the rightest decimals of number PI ?

But the "hash chains" solution, I really liked.
 
there are two issues. one, let us assume that people will try to game the system to maximize their odds. To minimize the effect of this, combining all the user created data to create a seed for a hash chain is quite effective. Probably if you said the first ticket was not the seed itself, but the first (or Nth) value from the hashchain, then this makes it quite difficult to control the bits.

Even if you can keep submitting tx offline until you get one you like. Especially if the starting value is 10 millionth!
so it is like the PoW for lotto tickets, if it takes minutes to calculate the lotto tickets and these are all very random, then it becomes impractical to gain an advantage by influencing your tickets.

The second issue is how to determine a winner. Each user now has a series of high entropy tickets, so it is just a matter of picking any sort of metric that is provable. I simply chose the distance function that I use for Telepathy, eg hamming distance.

You can use a much simpler "closest number" metric
or if everybody will have N tickets, you can use an N-dimensional distance

really as long as the method for calculating it is deterministic it becomes provable.

So, you need to protect the seed so nobody knows what it is, but to avoid you changing the seed to match the winning number, publish the hash of the winning seed. This way people can verify that you didnt change the lotto seed after the fact.

They would still need to trust that you didnt divulge it to someone else who would then have the advantage of being able to precalculate their tickets, but now we are getting to scenarios that require collusion, hashpower and tech skills

James
