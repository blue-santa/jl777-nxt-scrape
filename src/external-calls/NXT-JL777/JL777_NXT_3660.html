http://209.126.71.170/pNXTd is a prebuilt ubuntu privacyServer daemon. I have a couple instances running, but this is a decentralized system so the more the merrier and at some point it will be nice to have quite a few live to measure propagation times.

The source for pNXTd is at https://github.com/jl777/pNXT and you can build it on Mac or Ubuntu. If you have pNXTd running, then you dont need the thin client. This means the API is fully functional on Ubuntu. Of course, what is supposed to be and what is are two different things. Much chance for bugs.

So what is this InstantDEX release? Why is it pNXT, wasnt that a privacy thing?
Well, yes, but I said I was interested in the cryptonote tech for more than just the privacy. Now the cat is out of basket. I am using the peer to peer network for InstantDEX. I figure why reimplement a nice field tested peer to peer system when I can just tweak it for my purpose

So, the InstantDEX network will be riding on the pNXT nodes quite directly. However, considering the size of the download and the difficulty in building and the lack of 32 bit windows support, I realized a client/server approach was needed and that fits right into the privacyServer network I envisioned. Before you say aha! client/server, its centralized. I say, not exactly. Anybody an setup a privacyServer and it will be a bunch of servers each with 1 to N clients. Think decentralized hubs and some centralization around each hub. Basically the exact infrastructure we need for TF.

So, when you run the InstantDEX_t00 (testnet for Mac) it will initialize in less than a minute and start in local mode as far as InstantDEX goes. You can do all the orderbook commands but it wont go to the network, think "broadcast=false".

I have implemented a universal orderbook system that automatically creates new orderbooks on demand. Each orderbook is between two assets, assetA and assetB. Pure NXT will be considered asset #5527630, kudos if you can figure out why I chose that as the asset number for NXT. A little puzzle fun

So, the first step is to create an orderbook. You will notice on the API page there is place ask and place bid and they both have assetA and assetB fields. Most all the time you will just put in the orderbook id so the asset fields will be blank, but at first when there are no orderbooks we have to jumpstart it. I dont enforce the error checking but assetA < assetB is how it will end up, so if you put in backwards it will flip it on you so dont complain if things are backwards from what you wanted.

After you submit the place bid/ask, you should get back a JSON with the orderbook id. If you forget, you can just use the get all orderbooks API to get a list of all the orderbooks. When creating an orderbook, the actual order values are ignored so there wont be any active orders. You can verify this with get orderbook API. Notice there is a polarity field, -1 will flip things around, so the orderbook that you get will be reversed, eg. NXT/BTC or BTC/NXT depending on polarity

Now the fun begins. You can put in bids or asks in either positive or negative polarity. A bit complicated, but I think I got it all right, please verify as it is very easy to make mistake with this sort of thing. Everything is just numbers that all look not so crazy.

Once you start getting bids and asks, the get orderbook API is returning much more interesting things. If you set "allfields" to 1, it will display more info on each order, for now just the NXT address of the one who made the bid/ask. The orderbook JSON should be sorted in the right way, eg. decreasing for bids and increasing for asks.

Finally, I explains the specifics on how to put in a bid or ask. You need to orient yourself as to the polarity of the orderbook, its the difference between 12000 and .00008333. I decided to use floating point for inputing the price and volume. The display resolution goes to millisatoshis, eg. 11 digits, but internally it is good for a few more so I dont expect there to be much millisatoshi gremlins, though on windows machines anything is possible. So you can put in bids in prices at millisatoshi resolution and the volume has satoshi resolution. I think this is a pretty good range for trading. Now do not worry, the actual order gets converted from this price + volume format to unsigned 64 bits of satoshi precision for each asset. This is unlike how NXT does it where you are limited in digits for either the volume or price. With InstantDEX you dont have to worry about the decimal places an asset has. Just put in the exact amount you want with a millisatoshi price. I have not decided how to deal with rounding issues of assets without enough decimals. Maybe I will keep an internal tracking or maybe not. For now, I ignore this, so while it is true that the orderbook could display prices that are a bit different than what you put in due to (price + volume) -> (amountA + amountB) -> (price + volume), it will be interesting to see just how big this "problem" is.

Once you are comfortable with the API functioning, then you can select a privacyServer. For now use 209.126.70.159 unless you got one of your own running, then just put the IP address of your server in the "select my privacyServer" form on the :7777 page. this is the true test of decentralization, to see if the orderbooks synchronize with multiple servers. Once you are connected to a privacyServer, your API requests are routed to the privacyServer and broacast to the network. The responses you will get back should be quite similar to local mode, though the txid's will be quite different. Also, you might not need to create orderbooks, but for now I am not even verifying the asset ids are valid, it is all just input testing.

I dont have the ordermatching done yet, but just getting the orderbooks synchronized in less than 5 seconds (I hope) is a key achievement, especially with a decentralized topology. For now, please be nice to API, not much error handling so if you do bad things, maybe even crash, not sure. All bugs are important though.

I will try to get a windows version of InstantDEX_t00 out tomorrow, but I really dont like doing windows stuff, maybe I go to dentist instead.

I still need to deal with the rounding of assets without enough decimals, verifying assets exist and of course the actual trade matching. There is quite a lot of new stuff here, so please test thoroughly.

James
