nxt doesnt have any mixing at the moment meaning that if you sell illegal things TPTB will probably be able to track you down through blockchain analysis.

Let me put this straight.
Its not just about selling illegal stuff. When you are selling things you don't want others to know about your turnover nor who your customers are (with NXT they could even be contacted). With the blockchain explorer everyone out there can see your transactions.
My NXTmixer is based on the idea of a shadow account. Actually two, one for receiving and one for sending. These account numbers are only sent under encryption to the NXTmixer so nobody else will even know you have these accounts.

Now, all spending instructions from NXT accounts are sent under encryption, but everyone is always sending at the same time, so there is no timing analysis possible. People would tell the NXTmixer to send to your normal NXT account, but the NXTmixer reroutes it to your shadow account. You can change the shadow accts at anytime, so if you are receiving payments to shadowA, you now have funds that your can spend from shadowA and while blockchain analysis can find that payments came in and out of shadowA, it could be anybody's acct.

The problem is the initial funding of the shadow account. Unless there are strict restrictions on how much and when, knapsack analysis will be able to correlate funding of shadow accounts. This is where NXTcash comes in. The power of NXTcash (zerocoin) is that it can essentially teleport funds from one account to another (assuming you take proper precautions). However, the need to use a different IPaddress and preferably geolocation makes maintaining anonymity with zerocash type of solutions, not so user friendly.

My thinking is to use the NXTcash mechanism to fund the NXTmixer accounts. There is also the possibility of using AT to provide a funding mechanism. All of this needs to be wrapped into a user friendly interface that makes it push button easy and to enforce best use practices. Realistically, we are looking at summertime before there is any chance of getting all of this into place. I also have not figured out how exactly to deploy this, but it is actively percolating as I work on the other projects, which frequently end up creating nice modules for the anonymity stuff to utilize

Multigateway++ (jl777) / NXTsync - peer to peer synchronizing files or memory buffers
« on: April 29, 2014, 02:06:01 am »
For most of my projects, I kept wanting to be able to just map an in-memory data structure from another NXT node (with its permission!) to a local buffer. Without having to deal with sockets, TCP, UDP, etc. Well, I searched but couldnt find anything that quite fit the bill

I also ran into the problem of non-servers just not accepting connections, so peer to peer comms becomes problematic as one side has to accept the connection from the other. TCP is a lot more problematic than UDP, but UDP is unreliable. NATS, big complicated network stuff. Luckily I discovered the right search term for this problem "hole punching" and soon found a decent C library that did the basics.

This whole UDP hole punching is pretty tricky, but I think I finally have it pretty stable. It now has a chat system, encryption of most comms and strategic use of NXT token authentication to prevent spoofing. The encryption is using a different Curve25519 and a new public/private key is generate each time you start the program. less data for the bad guys to correlate.

Then there is the issue that UDP is unreliable and I can confirm that it is! OK, no surprise there, so round two added crc32 to each fragment and in the process of feeding back the crc that the receiving side got to the sending side. Not so easy though, because neither side can be sure what state the other side is in, so maybe send back the crc to let the receiver know that the sender got the crc that the receiver got. Who's on first??

I've spent so much time on this as the ability to easily synchronize data between arbitrary peers is pretty fundamental. Add in the fact that it is encrypted and happens in realtime and I thought it merited its own thread.

It will used by multigateway for withdrawal synchronizing, NXTsubatomic/NXTatomic for trade negotiation, NXTorrent for buyer/seller encrypted chat, etc.

Ive already got the flow control done and should be able to finish the reliable UDP layer tonight and hopefully then the synchronization will go quickly.

If all goes to plan, tomorrow I can port NXTsubatomic/NXTatomic to directly use NXTsync and maybe even code up the API. Then a NXTorrent update and back to multigateway. I need to finish that in time for NXT AE going mainnet!

marcus03:
I remembered that CfB and BCNext were monitoring the NXT network when we were under DDOS and asked him how they did it. Here's his answer:
"We had nodes in different datacenters. All these nodes were connected to Nxt network (to different peers!) and had direct connections to each other. Once a node received a block or an unconfirmed transaction it sent this to other nodes (not Nxt peers). The others measured time between the signal and time when they received the same block/transaction from Nxt peers. The delay is the main indicator that shows time of network convergence. During DDoS attacks average convergence time becomes much higher. Forks r also easily detected.
We also measure load on other peers by sending transactions to one of them from one node and receiving on another one. U can easily extended this approach and add extra functionality."
******

This is the basic approach I will use. I dont see why it cant be decentralized by each node direct connecting to random (or strategically selected) peers and then each subset cross reporting to the other subsets. There wont be any centralized point, the NXThealth monitoring would essentially be a parallel network.

Since all the code will be distributed across many nodes, an attacker would have to successfully attack the NXThealth network at the same time as the actual NXT network to pull anything off. If you are still paranoid, you can write a network correlator to make sure the NXThealth network is reporting the same data as the NXT network itself

I am currently testing NXTsync that allows any NXT node to directly sync memory (or files) with any other node. By default all comms are encrypted with a session specific key, additionally NXT token authorization can also be used.

Once this works, I think I just need to have each node update the sharedmemory with the raw data and update summary data, which then propagates to the other nodes. Just need to figure out what data each node needs to maintain. Probably just start with time of most recent block and its hash. Each node will end up with a slightly different view of the network. Once the NXThealth nodes are maintaining this data, each node can query a random sampling of the entire population of NXThealth nodes and as long as there isnt a big change with one subset vs the other subsets, it seems pretty safe to assume that is the real state

InstantDEX will let people change their bid/ask without fees and only charge when a trade happens. The goal of InstantDEX is to provide a fully decentralized peer-to-peer realtime trustless trading environment. There is no guarantee that this ideal goal will be 100% achieved, but I will continue to innovate and improve the tech to get as close as possible. I believe that it is important to provide the market what the market wants, instead of forcing something against what the market wants. To that end, InstantDEX will allow for trading of any NXT asset, NXT, bitcoind crypto against each other. I will also test market using ripple IOU's for USD and CNY to see if such a proxy of a proxy will be acceptable for traders. If it is, then it will be fully integrated into InstantDEX

Wesleyh will work hard to make InstantDEX as easy to use as possible.
