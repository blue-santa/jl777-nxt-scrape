I think you are extrapolating from an assumption and then deducing things from that. It's not ideal.

Some more information to allow further analysis:

The Lchain is isomorphic to the blockchain

L0 is the empty ledger (genesis)
L10000 is the ledger at block 10000

L0 + blocks 0 to 9999 -> L10000

Part of the ledger is the ledgerhash which is comprised of 11 sub-hashes (one for each dataset)
Extrapolating this, we get L10000 -> L20000 -> .... L400000

With the property that you can start from any ledger and replay the blocks between that ledger and the next one
and end up with the same hashes (still some bugs I have here, only half the hashes match in the resume case)

Now each node is keeping their Lchain hashes and a new node can query N peers to verify that there is consensus on this. it can also reprocess from genesis if it wants

Since there will be 100% consensus on older blocks there is no room for any mismatches on the ledger hashes, until we get close to realtime. But, if you notice, the time consuming part of the process (recalculating from genesis) has no controversy and you can quickly find the only valid sequence of Lchain, get a version from a week ago, verify its hash.

and then just reprocess from there to realtime. Now this approach assumes there is still a blockchain for recent blocks. The key is to offset the latest ledger to far enough in the past so that a false chain cannot overcome the valid chain weight. something like every day, generate a ledger for a week ago. Then the only blocks ever needed are the past week's blocks. These timings can easily be tuned. Also, it is assumed that there will be some archival nodes that are storing all the historical blocks, but after all the nodes are validating the Lchain and if 100% of the nodes are showing the same ledgerhashes, then this is not needing any decisions by new miners on which fork to use as there wont be any forks for the past.

Hybrid solution. Use ledgers for the past where there is no forks, use blockchain for recent blocks where things are still in flux. By combining these two methods, I expect the Lchain to grow linearly in size as opposed to exponentially.
