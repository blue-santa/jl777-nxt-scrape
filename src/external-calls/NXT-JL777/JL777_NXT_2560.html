General / Re: 2-Phased Transactions Pre-Implementation Specification
« on: August 26, 2014, 06:03:29 pm »
Please add MinHeight, so that a transaction cannot happen before a certain height. This is useful for making microchannel payments, but we would also need a way to cancel it under right conditions.

The idea is to have an equivalent for: https://bitcoinj.github.io/working-with-micropayments

The key is to lock in a payment to yourself, but this is in the future and it is an insurance against not receiving payment
once this is in place, then a multisig acct is funded. Now these funds cannot be accessed unless the other party approves it.

Now we dont want 100 tx fees and the blockchain clutter broadcasting each micropayment, so this is done offchain using a properly signed tx. At this point the destination account can cash it in at anytime and that is OK, but then it needs to come from the same funding source as the insurance payment. Each "payment" is simply sending a signed tx with a larger amount.

I have implemented synchronized and overlapped micropayment channels between two coins (any bitcoin fork), so each side makes sure if it blows up, they can get their funds back with a future payment to themselves. Then one side sends a small amount, the other side matches, repeat until all is transfered. then each side finally broadcasts the final (largest) payment and both sides are happy. Neither side ever took more than 1% (for 100 payments) risk. In the event things get aborted, then one side is out 1%, not great but viewed as escrow fee actually within reason. So the worst case is you pay normal escrow fee, the expected case is smooth exchange. In the event no payments are even sent, then when the minheight is reached, you just collect the funds that were held in reserve.

It would sure be nice to be able to use this same mechanism so that I can do subatomic exchanges of NXT against the bitcoin forks!

James
