So it still gets back to needing to look at the stratum protocol to see what is exchanged between server and miner.  The MS minter needs to vary the nonce in the 40-byte hash data while the Bitcoin/Altcoin miner varies the nonce and additional fields in the 80-byte block header.  This is done by the miner for each hash iteration.  So my guess is that something like BFGMiner will not work for MS coins since the data format is completely different.

Nxt Monetary System / Re: We need GPU miner
« on: January 16, 2015, 06:47:31 pm »
Quote from: ScripterRon on January 16, 2015, 04:38:16 pm
I'm currently testing a stand-alone minter that I wrote in Java.  It supports all of the minting algorithms for CPU hashing and has a GUI which displays transactions minted during the current session.  It handles the duplicate transaction problem by waiting until the block height increases before submitting the next transaction (the transaction is queued and minting continues while waiting for the next block).  You can still get a failure if you stop and immediately restart minting while a transaction is pending since the program will no longer know about the pending transaction.  If this becomes a problem, I'll look into retrieving unconfirmed transactions from the server and checking for duplicates during initialization.

NxtMint (the minter) uses NxtCore (common API services) to provide the NRS server interface.  This means that minting transactions are created and signed locally and then broadcast to the server.  The secret phrase is never sent to the server and a rogue server cannot create a bogus transaction.

I've been testing it on Windows using EGOLD.  I get around 1.3 MHash/s for each CPU thread (i7 processor)

I've also written GPU support for the KECCAK25 algorithm (the algorithm used by EGOLD).  This gets 12-16 MHash/s (NVIDIA GT 720) depending on the number of parallel instances.  Note that increasing the number of instances can cause the video performance to suffer, so it is best to use GPU minting on an otherwise idle system.  The number of open GUI windows also impacts GPU minting since it reduces the GPU resources available to OpenCL.  You can run a mix of CPU and GPU workers with the trade-off that increasing the CPU load will begin to impact GPU processing since Aparapi/OpenCL need CPU resources to manage the GPU instances.  I've run with 4 CPU threads (on an 8-processor system) without impacting the GPU.  If GPU execution fails, the GPU worker will revert to CPU hashing and keep running.

The GPU support uses Aparapi from AMD and OpenCL from the card vendor (NVIDIA in my case).  I'm sure performance can be increased even further by writing a native C/C++ minter, but I prefer to write in Java.  And Aparapi was an interesting project since it translates Java bytecodes into an OpenCL program (you write your GPU kernel in Java and Aparapi handles it from there)

I'll release it on GitHub later today after I let it run some more.
