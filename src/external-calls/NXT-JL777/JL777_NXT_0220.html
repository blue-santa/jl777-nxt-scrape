General / Re: Do sent transactions reuse R values as in bitcoin?
« on: August 04, 2015, 06:36:58 am »
Quote from: doctorevil on August 04, 2015, 05:55:10 am
Quote from: monsterer on July 28, 2015, 09:59:18 am
Firstly, apologies I must confess my ignorance - I'm not sure if the same crypto is used in NXT to generate signatures from private keys as in bitcoin.

My question is, do transactions sent from a single account reveal partial information about R values from the private key like transactions sent in bitcoin would if you reused the same private key?

I'd like to read more if anyone has a reference.

The short answer is no.

The longer answer is that it's complicated. 

Bitcoin's signature scheme is ECDSA, NXT uses EC-KCDSA.  As originally specified both algorithms require the signer to generate a unique random value k (from which the "R value" is derived via a one-way-function) as part of the signature.  Using the same k for two different messages leaks your private key.  Using a k someone else can predict will leak your private key.  At some point cryptographers realized this situation royally sucked.  If users constantly fuck up providing enough entropy during key generation then requiring users to supply security-critical entropy every time they sign make the situation worse.  So people came up with general techniques to "derandomize" signature schemes. NXT's EC-KCDSA implementation uses derandomization, but there isn't anything stopping you from "going off the reservation" and writing an old school "randomized" EC-KCDSA implementation because from a verifier's perspective "randomized" signatures are indistinguishable from "derandomized" ones.

Further reading:
[1] https://tools.ietf.org/html/rfc6979
[2] http://blog.cr.yp.to/20140205-entropy.html
[3] https://gist.github.com/doctorevil/9521116
[4] https://en.wikipedia.org/wiki/Elliptic_Curve_Digital_Signature_Algorithm

by using only pairwise shared secrets, I got a curve25519 chain to generate the identical result from different privkeys with the restriction that only the pairwise shared secrets are used. assuming there are no exploits for this, then an arbitrarily large mixing set can be used without needing any coordination between the accounts.

       (curve25519(curve25519(curve25519(curve25519(A,curve25519(B,c)),curve25519(D,e)),curve25519(G,f)),curve25519(X,y)));

the weakness with the bitcoin was that there is an math equation that uses the privkey and a random value to create what is publicly disclosed.

so with bitcoin it depends on a math equation that can be reverse calculated directly and the only protection against that is that a high entropy R value is used.

with NXT SHA256 is used and it has no math equation to be reversed
