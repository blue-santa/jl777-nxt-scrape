Had to deal with a few bugs, but here is a new version:

http://209.126.70.170/uMGW_v2

It has far fewer printouts so you can see what matters a lot better
I debugged it with BTCD all the way to the withdraw rawtransaction signing. Seems there is some sort of issue within BTCD, but as far as MGW goes it is looking pretty good.

I created two new JSON fields to support BTCD: "useaddmultisig" and "nohexout"
If your coin doesnt have a "createmultisig" command, then set "useaddmultisig":1
If your coin doesnt put the "hex" output in the txout, then set "nohexout":1

The way to tell whether it supports createmultisig command is to go to the help meny, debug console and type in help
If the list has createmultisig, no worries, if not, then hopefully it has "addmultisigaddress" If neither, then MGW wont work with it.

The second one is a bit more challenging to find.
First get a txid, any will do. and do:
getrawtransaction <txid>

This will spit out a bunch of bytes, take those bytes and do:
decoderawtransaction <a bunch of bytes>

Now you will see a raw transaction in human readable form, find the scriptPubKey entry for any txout:

"scriptPubKey" : {
"asm" : "03fffbcc77e41d62f9bbd4843eae645671df5a7fa1c9393b9e2e03dd3e0312a537 OP_CHECKSIG",
"reqSigs" : 1,
"type" : "pubkey",
"addresses" : [
"RBCLnsjCVHWRyUpfgauyzUu8iP6X5CabJT"
]
}

Notice is has an "asm" field, but no "hex" field, so if your coin is like this "nohexout":1 is needed

James

P.S. for reference here is what it looks like if it has "hex"
"scriptPubKey" : {
"asm" : "OP_HASH160 7561bf1bf097cc4776e838c57dfb5be4f67c94cc OP_EQUAL",
"hex" : "a9147561bf1bf097cc4776e838c57dfb5be4f67c94cc87",
"reqSigs" : 1,
