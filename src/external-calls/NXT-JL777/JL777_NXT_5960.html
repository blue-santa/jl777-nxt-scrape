Nxt Community News and Announcements / Re: Development roadmap

Hope its ok to post my roadmap here:

multigateway: (BTC, LTC, DOGE, DRK,...) <-> NXT AE
BTC bridge: automatically converts NXT -> BTC -> any BTC address so NXT is accepted everywhere BTC is!
NXTprotocol: websockets interface plus eventually dynamically loadable/unloadable handlers
nodeminer: will get some cool features and ways to earn more than just nodecoin
NXThealth: monitors NXT network status, probably will get integrated into nodeminer
fiat support using Ripple
NXTdividends: sends out dividend proportionally to holders of an asset
InstantDEX: realtime trading of any NXT asset for another other NXT asset (or NXT)
NXTdatafeed: prices and other data compiled from external sources and published on NXT blockchain
NXTbetting: make bets based on NXTdatafeed

NXTmixer: nacl encryption based and also avoids timing attacks, the fact it was initial funded cannot be avoided, unless...
NXTcash: allows initial funding of NXTmixer anonymously

Multigateway++ (jl777) / BTC bridge: buy anything with NXT that you can buy for BTC!
« on: April 12, 2014, 02:49:34 pm »
My plan is to add support for a destination address to the multigateway withdrawals (see C below). The comment field allows me to bind a specific redemption request to an external BTC address. Just need the GUI to make it push button easy. It will create a token to verify that the NXT acct holder is the one doing the transfer. If they can create the valid token, they can just as easily just transfer out all the NXT.

With the above, there are still three things:
A) Availability of BTC asset at decent price
B) NXT -> BTC asset
C) BTC asset -> multigateway -> destination BTC address

Maybe A) wont be a problem at all as BTC will probably be a pretty popular asset. If people also knew that simply placing NXT AE orders for NXT -> BTC will support the BTC bridge and enable spending of NXT anywhere BTC is accepted, they might be more inclined to provide the liquidity.

I could create a multigateway function to combine B) and C), but ideally it would need a comment field added to SendMoney API so I can do it seamlessly. There should also be a little GUI support to ensure that there is enough liquidity at the moment and to display the actual cost in NXT for the BTC.

If there isnt enough liquidity, I do have some ideas on how to automatically create some, but it would require punching through orders to a centralized NXT/BTC exchange, with the assorted issues that raises.

General / Re: Simple escrow for asset exchanges
« on: April 11, 2014, 08:49:17 pm »
Quote from: Jean-Luc on April 11, 2014, 08:23:36 pm
Quote from: dmk on April 11, 2014, 07:11:43 pm
The fundamental point of what I was talking about is that the NXT blockchain itself would be used to guarantee that A and only A paid B a specific amount and that B and only B gave a specific amount of assets back to A as a result.  If either A->B or B->A doesn't happen, then neither part is validated and no one with a copy of the blockchain could dispute that conclusion.  I can't see how AT could solve that problem without in some way modifying the blockchain to contain this type of "paired transaction ledger" information.
Right, it would be better if a dependency like this could be defined and enforced in the blockchain itself, without relying on AT.
Atomic swaps of X amount of assetA for Y amount of assetB would be fantastic
I really need this ability and even if it is limited to just A <-> B with different signers for each, that would enable totally decentralized exchanging.

What do you think of the following method:
A) both parties create a token where the "website" is JSON that describes the trade, something like:
{"time","<timestamp>","accountA":"<NXT addressA>","assetA":"<assetidA>","quantityA","<quantity>","accountB":"<NXT addressB>","assetB":"<assetidB>","quantityB","<quantity>"}
B) A special transaction that accepts either accountA or accountB's passphrase as long as the two tokens are present (or the token of the one not signing).

A little cumbersome, but it will all be program generated and I am sure implementation will be much simpler than a super elaborate system. The timestamp is necessary to prevent double spends and both parties would need to agree to the exact time stamp.
