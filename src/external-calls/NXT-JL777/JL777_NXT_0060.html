SuperNET Releases / SuperNET_API chrome app and standalone version in one release
« on: November 02, 2015, 12:55:13 am »
I ported SuperNET and nanomsg into pnacl, which allows it to be a chrome app. I then back ported this into a unix build so the same code can be run natively in unix (and osx) as with the pnacl in any chrome browser
it would be possible to port this for windows, the makefile.unix shows how to build it for native use.

i also got all the secondary instantdex API working for the 9 exchanges, to the extent the exchange actually supports it: "placebid", "placeask", "balance", "openorders", "orderstatus", "cancelorder", "tradehistory" and "withdraw" works across bitfinex, bittrex, btc38, huobi, lakebc, okcoin, poloniex and quadriga. all using the same request syntax. the "balance" api can specify "base" to get the balance for a specific coin.

git clone (or pull) https://github.com/jl777/SuperNET_API
cd SuperNET_API
./m_unix will build SuperNET, SNapi and BitcoinDarkd

SuperNET is the standalone, SNapi allows to submit API requests directly and BitcoinDarkd is a combined version

alternatively, you can run tools/httpd.py -C . -p 7777 &
and go to http://localhost:7777 where there is an ugly gui where you can submit via JS calls the SuperNET requests. SNapi will also send requests to the chrome app.

There is an index.html which can be modified to create a custom application that directly calls SuperNET_API and there is no other anything that is needed, at least it can be configured that way. the following are the specific InstantDEX API calls I tested

I have been thinking about the same and have a friend who is Java dev looking at this. (I have been working on the App together with James who did the great NaCl porting)

To be honest google Chrome Apps are not yet very well documented (they have somehow covered anything somewhere but the structure is not very well placed yet and the videos they make are kinda horrible). Seems like most of it came from 2013 and it kicked off nicely and they make slowly progress with very great outcome. So it allows to have SuperNET (as C based) to be available as an App, which is awesome.

Yet, it is strongly sandboxed (what adds a great security). So including Jay will need a bunch of tests and in the end straight setup which we need to test within the next weeks. (this enables all Nxt functions with connecting to public nodes but not being a Nxt host within the App)
The problem is that every IP you connect to needs to be defined. The google App wants to know (and it is public to the user for security reasons) where you get your data from.
I cannot yet say if it is possible to have Nxt running in such an App. Also the Blockchain is another issue, you would either need to have a centralized point of download (which makes it centralized and we would want to find decentralized solutions) or use the "slow" peer-to-peer approach again. Then instead of finding a solution to the App approach, Nxt itself is simple enough to install for itself and we would not need an App...
InstantDEX and peggy are getting quotes from external websites, this problem is not a problem. We are working out the details of allowing the C code to make Jay requests, which would allow for all the lite wallet usage of NXT from the chrome app.

I do have all the bitcoin and curve25519 encryption code in C code though, so if needed I will be able to do all the crypto calcs in the C code
