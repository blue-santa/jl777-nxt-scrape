90  Alternate cryptocurrencies / Announcements (Altcoins) / Re: [ANN][KMD][dPoW] Komodo - Zcash Zero Knowledge Privacy Secured by Bitcoin on: March 24, 2017, 09:43:05 AM

kpx.io is on the mainchain and it is advancing, slower than normal until we get the GPU pools back online.

Only the beta branch is up to date for now as we are still verifying that this specific attack is fully solved. The usual "after the storm" advice of dont do large transactions, dont worry if basilisk is slow/not working, wait for updated GUI, etc.

Description of attack:

There are three (actually four) different contexts in the bitcoin code of validating a transaction. KMD verifies that the nLockTime in the tx is not too far in the past. A one hour leeway was selected as this amount of accrued interest is lost to avoid the third attack we had the other day.

When the miner creates the block, when the block is relayed and extends the chain tip, and when the node verifies the local blockchain. All of these are clearly required and need to be 100% deterministic in the value used. There is even a fourth validation, of wallet transactions.

Due to the quick work required to stop the third attack, I missed that there was not 100% agreement among all the cases. If it is even off by a second, it would be possible to fork the network. The difference was on verification of the blockchain a node does on resuming/starting sync. (Oh yes during initial sync is a fifth different context!)

The attacker started spending some of his illicit 1400 KMD he got from the third attack. Every block he was backdating it the maximum amount. At first we thought he was just trying to get some fractional extra KMD. But what he was doing was getting a tx confirmed that would be rejected by the resume scan!

As soon as he achieved this, he started a DOS attack sending malformed messages (details of which I havent had time to analyze) that crashed nodes, forcing them to restart and reject the main chain. Each node would then be on their own and if it was a mining node it would start its own fork.

The first thing I did was stop the crashing with a patch so we could assess what was going on. Then we figured out the bug with the timestamp. Then made a few (a lot!) of iterations on possible solutions, finding out more and more different contexts. Finally when the five contexts were understood, I was able to construct a solution that was consensus safe.

At least that is my current understanding, which we are verifying.

Unlike the other attacks, this attack did not seem to be financially motivation as there was no coin creation issue. As I said before more attacks are likely to happen. We will continue to combat it as best as we can to minimize service interruption. With each attack we are getting better at damage control and resumption of the chain.

The first priority is to protect your KMD. Currently we are a bit diff stranded, so until the GPU pools get online (using the beta branch) the blocks will be really slow.

As I wrote before there is now a one hour timeframe for a tx, so there is a compare against (timestamp - 3600) to check for tx nLockTime validity.

This one hour of leeway does allow collecting one hour's interest, but at 250000 we will subtract 59 minutes from the effective time. So this free interest will close soon.

We saw that the attacker was taking advantage of this, which made little sense as it is only getting small fractional KMD. Until we realized it also had the effect of creating edge cases, which would make a node create a block that it would then reject on restart.

The error is totally my fault, I used the chaintip's timestamp instead of the block's timestamp, not even sure why.

Now to make things more interesting as things would be so boring with just an issue like the above, there appears to be a DOS attack with killer network messages that crashes nodes randomly. Not sure if it is an intentional thing, or maybe it is just due to the current chain's state.

Anyway, I put a patch in for the crash, so it stops crashing now, but the cause of it is not time enough yet to solve.

I also pushed a version that uses the consensus safe value for the timestamp check. Notaries are in the process of verifying this.

From what I can tell the vast majority of the network is on the same chain, so we might not even have a fork, but with nodes randomly crashing it is hard to say. When the crash resistant version is propagated we can better assess things.

The black hat made something like 1400 KMD from his attacks. He could make far more if he just reported these issues...

Anyway, it is very good that we are getting help from very capable people to help solve these issues in a timely fashion. Thanks to all!
