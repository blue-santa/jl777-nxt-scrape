This error is related to minting transaction validation. The protocol enforces that each minting transaction hashes an ever increasing numeric counter value per account/currency combination in order to prevent minters from simply submitting the same solution again and again.
Therefore when a client submits a mint transaction the NRS compares the counter to its database records to see if the counter is larger than the current counter stored in the database. If it is, then the transaction is accepted, if its not then it becomes interesting, since this transaction may still become valid when the block containing the other mint transaction with this or higher counter is popped off.
We call this type of transactions "not currently valid transaction" and normally such transactions like, for example, payments without sufficient balance, stay in the unconfirmed pool until either they expire or some condition changes and they get accepted.

The problem with mint transactions is their popularity, in the early days of MS and perhaps even now, minters submitted 1000's of mint transactions with incorrect counters. These transactions flooded the unconfirmed pool and slowed down legitimate transaction propagation. 
Therefore since 1.4.8, only for mint transactions, we reject the transaction immediately in case the counter is not increased so it does not fill the unconfirmed pool. The weakness in this approach, is that different peers has different set of transactions at any given time,  therefore it is possible that some mint transaction is accepted by one peer but rejected by another peer. One of these peers is left on a fork. Moreover, it looks like in your case, not only that your peer is on a fork, but since the mint transaction it rejected, was accepted into a block by all other peers, your peer is black listing the rest of the peers.

Solutions:
1. Re-download the blockchain.
2. Popoff all blocks up to the MS block and let your node reload the rest.
3. Identify the account and currency with counter set to 45 and using SQL reduce the value to 44 in the currency_mint table

Jean-Luc feel free to review my answer to see if I missed anything.
This is not a long term solution.
If there is no way to avoid this, we are back to having to rescan the blockchain (always at the most inconvenient time).
