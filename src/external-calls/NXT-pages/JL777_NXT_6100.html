Quote from: Jean-Luc on March 29, 2014, 09:53:39 pm
The problem is not that you cannot get the transactionId or hash before submitting the transaction. But that the id (or hash) themselves are calculated as hashes over all of the transaction bytes - including the referenced txid/hash. So the id of transaction A depends on the id of B, and vice versa. And if you exclude that field, you can no longer uniquely identify the transaction.
Maybe I am getting enough imagination.
I think I have a solution.

If we had a new field for transactions (i know big change) that specified the block number that it executes at, then all these dependencies can be solved.
We also need a new command that allows changing the execution time with reftxid

It will also allow for many other uses.

submit txA due to execute in 10 blocks
submit txB due to execute in 10 blocks
submit a replace execution time with reftxid txA with txB as reftxid
submit a replace execution time with reftxid txB with txA as reftxid

Now the first two transactions are cancelled and the two active ones are the replace execution times. I think all the data needed for proper hashing is available for the replacement tx's. We still have the two block time to establish this cross trade, but this seems to be a lot less work than implementing arbitrary orderbooks

I narrowly defined the "replace execution time with reftxid" command to specifically address this use case. With a bit more imagination, it should be possible to address a far wider range of use cases and allow us to get semi-smart contracts much, much sooner

James
