« on: June 09, 2014, 02:44:31 pm »
Quote from: danp123 on June 09, 2014, 02:31:21 pm
This is for the asset exchange api.  Why are these numbers 10k off or so?  I think it's the number multiplied by the total amount they are selling, but how do I get that on the API?  Do I have to look up each sell order?

I found this:
"Since block 134,000 NXT is divisible up to eight decimal places. NQT, or nxtQuant, is the temporary name given to 0.00000001 NXT (or 10^-8 in mathematical shorthand)."

But still it makes no sense in the data I'm getting.
for each asset you need to get the number of decimals, for NQT is it 8 decimals
What I do is I "calculate" a multiplier based on the number of decimals

        for (mult=1,i=7-ap->decimals; i>=0; i--) // done at init time
            mult *= 10;
        ap->mult = mult;

Now you can convert all QNT to satoshis by multiplying: (QNT * ap->mult)
To display in user friendly fashion I have a dstr(x) macro that just does a ((double)(x) / SATOSHIDEN) where SATOSHIDEN is 100000000 so I can use %.8f and the user sees a 1.23456789 number instead of largish scary looking numbers. Note that for assets with less than 8 decimals the least significant digits will always be 0

so an asset with 4 decimals will end up with 123450000 -> 1.2345

hope that helps. You do need to verify each and every asset handling to make sure the decimals are handled correctly.

getbalance now becomes the sum of all NXTaddresses associated with the account
getinfo, getblock, getblockhash, gettransaction all have 1:1 mappings, just a horrible headache matching up the NXT info into the bitcoind format

listsinceblock becomes a matter of iterating all the blocks since the specified one and using logic similar to getbalance

So, this list of commands comes right up to the point of being totally horrible, but not quite. They are not needing anything that actually needs to address the individual txouts, though listunspent comes very close to this. If they start needing to use the output of listunspent to create tx with inputs, then without simulating txout/txin threads on top of NXT accounts where everything is fungible, will yield unpredictable results. As long as they are using listunspent to just get available account balances, it should be OK.
