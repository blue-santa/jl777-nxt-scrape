Thanks. AFAICS the function generateToken already mixes with a local timestamp, see [1]. A very neat way of using a TTL mechanism would be a recent blockhash. The general problem of PKI of course is knowing that example.com belongs to me. But as I'm using not the web/DNS it is be sufficient to register an alias. porting old names from the real world or standard DNS is impossible. I think with blockchain based authentification other salting becomes unnecessary.

[1]
https://bitbucket.org/JeanLucPicard/nxt/src/9b0ffb406e08cc7e8aed5161f362b2d1640c8647/src/java/nxt/Token.java?at=master#cl-17
yes, I think it does change with the time as you get different token for the same string each time you generate a token. My point is that ALL these tokens are valid. And I believe they are valid forever. So what is to prevent someone from getting a token and using it over and over and over?
If the token enabled a money transfer or even just something that costs anything, an acct can be drained by iterating the action over and over.

What I do is I put the current timestamp right inside the critical requests and then on decoding I verify that the token was made recently (sometimes as short as 3 seconds). Basically this implements expiring tokens.

Now if you just want to verify access to some read only data, this is probably overkill.

Account Control / Re: Account verification protocol
« on: June 10, 2014, 04:48:34 pm »
Quote from: benjyz on June 10, 2014, 04:13:00 pm
I have implemented an account system using a simple Nxt Auth protocol and ZeroMQ (not really relevant here, but they have their own Auth libraries and way of doing messaging networks). It works as follows. I would appreciate some feedback.

The client requests access with one message "AUTHREQ", providing a generated token for the website, as well as the accountID (as RS format):

Code: [Select]
String SECRET = "secretpass";
String WEBSITE = "example.com";
String token = NxtAuth.genNxtToken(SECRET, WEBSITE);
System.out.println("token for the website: " + token);
String myAccRS = NxtAuth.getAccountIdRS(SECRET);
String delim = "|";
String req = "AUTHREQ" + delim + token + delim + myAccRS;
The server verifies with:

Code: [Select]
boolean tokenVerify = NxtAuth.verifyNxtAuth(WEBSITE, token);

String accountShould = NxtAuth.decodeNxtToken(token,WEBSITE);

boolean accountEqual = accountShould.equals(acc);

if (tokenVerify && accountEqual)
   valid = true;

if (valid)
   reply = "ACCESS_OK";
else
  reply = "ACCESS_DENY";


This is a one time request. In my simple implementation the server adds the account to a whitelist and will accept all connections which provide the ID. This is vulnerable to a scan attack where a malicious attacker attempts providing any ID (which is currently okay for me, because the number of auth requests will be very low and I can block scanning). Another issue is that I store passwords in plaintext on the client side. I haven't looked into how Nxt handles this.

As far as I can see the token system uses a mix with timestamp data, so that one can implement a time-to-live (ephemeral keys).

I'm thinking about what kind of security such a system can be provide in comparison to the existing CA's.

Edit: What is missing is the full handshake (server => client). Presumable the website registers an alias? In some cases the website could also be just a shared secret instead.
I would include the timestamp in the token generation. Otherwise anybody that obtains a token could reuse it
Most websites just seem to validate the acct and dont bother with matching the exact string used, but for money sensitive commands I think requiring an exact match with a timestamp is recommended
