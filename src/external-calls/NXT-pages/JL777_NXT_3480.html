************* pre-release notes
I got tired of having to constantly fiddle with making new builds and also having multiple builds is not so easy to manage. Also, what about supporting new coins, changing servers, etc.

So, I added support for an MGW.conf file. Just have the file in the same directory as the MGW daemon and it will load the parameters. The file format is one JSON string, the following are the fields:

// if MAINNET is not specified or is 0, then it is testnet
"MAINNET":1

// if secret field is not there, then the randvals files will be used to generate secret
"secret":"<account passphrase>"

// Optional fields that will override the defaults:
"MIN_NQTFEE":100000000
"MIN_NXTCONFIRMS":10
"ORIGBLOCK":"<hash of first block to start scanning>"
"NXTAPIURL":"http://127.0.0.1:7876/nxt"
"NXTISSUERACCT":"<account that issued the MGW assets>"

// MGW Fields:
"GATEWAY_SIG":12345678
"SERVER_PORT":666
"MGW0_ipaddr":"12.34.56.78"
"MGW1_ipaddr":"87.65.43.21"
"MGW2_ipaddr":"11.11.11.11"
"special_NXTaddrs":["<MGW0>","<MGW1>","<MGW2>",...]
"active":["BTC","LTC",...]
"coins":[{<coin JSON>},{<coin JSON>}...]

The <coin JSON> is as follows, set NXTfee_equiv to be close to what 1 NXT is:
"name":"<abbreviation for coin>"
"asset":"<asset id for coin>"
"txfee":0.0001
"NXTfee_equiv":0.0001
"rpc":"127.0.0.1:666"
"conf":"<full path to configuration file for coin"
"blockheight":<first block to start scanning blockchain from>
"minconfirms":10
"estblocktime":300

// optional fields to override defaults, probable no need to change these
"nohexoutput":0
"useaddmultisig":0
"forkblock":0
"pollseconds":60

Once you designate a server as MGW0, then if that ever changes, all the deposit addresses have to change. By changing the SERVER_PORT and GATEWAY_SIG, you can run multiple MGW's on the same server and they will only look at the AM's with the matching signature. If you are testing with assets you issue yourself, you need to set NXTISSUERACCT. You can even use a remote server for API calls by changing NXTAPIURL, but probably not such a good idea. The ORIGBLOCK determines what block number NXTservices starts scanning the blockchain from. If you dont need historical data, just set it to a recent blockhash.

{
  "MAINNET":1, "GATEWAY_SIG":1234567, "SERVER_PORT":666,
  "MGW0_ipaddr":"209.126.70.170",
  "MGW1_ipaddr":"209.126.70.156",
  "MGW2_ipaddr":"209.126.70.159",
  "special_NXTaddrs":["423766016895692955","12240549928875772593","8279528579993996036","5723512772332443130","15740288657919119263"],
  "active":["BTC","LTC"],
  "coins":[
{"name":"BTC","conf":"/home/root/.bitcoin/bitcoin.conf","asset":"4551058913252105307","txfee":0.0001,"NXTfee_equiv":0.0001,"rpc":"127.0.0.1:8332","blockheight":300797,"forkblockheight":308539,"minconfirms":3,"estblocktime":600},
{"name":"LTC","conf":"/home/root/.litecoin/litecoin.conf","asset":"2881764795164526882","txfee":0.001,"NXTfee_equiv":0.01,"rpc":"127.0.0.1:9332","blockheight":564980,"forkblockheight":594965,"minconfirms":5,"estblocktime":150}]
}

You can download a Universal MGW from http://209.126.70.170/uMGW
It is only for Ubuntu and I dont expect this to change. Make sure to make a backups directory from where the coin daemons run from, usually something like ~/bitcoin/src/backups. Make sure to have the identical MGW.conf files on all three servers. The only thing that could change are the "secret" field, the locations of the coins .conf files and the rpc ip/port, everything else must match or the servers wont agree on things

So now, you can run one (or three) customized MGW servers to support your favorite coin!
However, only use with NXT is authorized, any other usage is likely to end badly.
The following coins are supported, but only the ones with (*) have been verified:
BTC*, LTC*, DRK*, DOGE, CGB, BTCD, VRC, PPC, NMC, XC, VRC, ZET, QRK, RDD, XPM, FTC, CLOAK, VIA, MEC, URO, YBC, IFC, VTC, POT, KEY, FRAC, APC, VOOT, GML, SYNC, CRYPT, RZR, ICB, CYC

OK, so maybe some of these coins dont support multisig or not quite compatible, but if it is a gen 1 coin, then there is a decent chance it will work out of the box. The further away from a pure bitcoin fork, the less likely. I still have 30 slots open and if you want to support a coin, then find two other like minded people, get a MGW.conf file that works, we can consider adding support for your MGW in the official MGW lite GUI. For a coin with low volumes, a VPS with 8 gigs of RAM should be adequate to be a MGW server. As long as you deal with issuing the asset and taking care of the servers, you can use it for free and have your favorite coin trade on NXT AE via MGW.

With the ability to change the API URL, it is quite possible that NXT clones could just use this as is. If you are a NXT clone maker and are adding value, then I am open to cooperation, just contact me. I have a few conditions for allowing MGW to run on a clone. My interest is in making a globally decentralized infrastructure, so as long as you agree to participate in that, then I will most likely allow usage of MGW. For non-NXT clones, this applies, plus some sort of reasonable incentive (compensation) for porting.

First off, MGW might be super important, but its "database" is 99% stored on the blockchain(s), so data loss is not a worry, basically all the important things are in the AM's and asset transfer transactions, so the entire state can be recreated. Actually any MGW client can duplicate this internal state if they hook up their bitcoind and this provides the realtime transparency of unspent outputs.

So, even if all three servers go offline, as soon as they are backup, they process the blockchains and catch up.

Now as far as the 1% that is not on the blockchain, it is the private keys to the accounts and I havent figured out a way to distribute this without compromising security, so each MGW host is responsible for making sure proper procedures are used for protecting and backing up the private keys.

Now I am no expert in running a server in Tier 1 status, but I have done three major iterations on the withdraw processing to boost its throughput to almost match the deposit side. On the deposit side, the throughput is only limited by the respective blockchains! So that is very redundant and high capacity. On the withdraw side, the batch processing combines all pending withdraws into a single transaction. So, while it can only do one per cycle, it can handle literally hundreds of withdraws all at once, so I doubt there would be any realistic case where the withdraws bottleneck things. At worst, it would just chug through one batch after another.

Remember that MGW only has to process deposits and withdraws. I expect there to be 10 to 100 AE transactions for each deposit/withdraw, so with the current MGW capacity I dont expect any issues.
