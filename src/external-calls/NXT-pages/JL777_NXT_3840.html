let me explain how the caching works
On startup it reloads all the .blocks and .cache files, so the smaller these files are the faster

now after a successful launch, it creates an ignorelist. If there is an ignorelist, then a lot less .blocks items are loaded, but all the .cache files are still loaded

During runtime, if it is a block not on the ignorelist, it loads it and all its tx's and saves them to the cache, so the amount of time you are running after the successful launch determines how big the .cache file gets. This means that if you have a recent ignorelist, you can delete the .cache file and it will only load the tx's in a block that is useful to the cache

I guess as long as you dont delete the ignorelist, you can probably get away with deleting the .blocks file, but it is handy to have a complete block cache and it is skipped loading into memory on startup so it is only using up disk space and not RAM

So, a pretty simple caching logic, but it ends up with somewhat complex behavior. To keep it simple, just delete the .cache file at the beginning of a day where you will be doing many test launches

2. Wait until the batch withdraw is seen on blockchain
3. back to 1.

What this means is that if you submit 10 withdraws all to the same address, it will get combined into a single output in the batch and that will be within a single batch tx.
