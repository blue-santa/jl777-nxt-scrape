I think the interfaces are pretty close. A few tweaks to make it able to just drop-in.

I have all the contract pairs specified as base/rel (baseid/relid)
where base -> rel at the specified price. with base NXT and rel BTC -> NXT/BTC

this allows the user to flip the display to their preferred orientation just by reversing the base and rel. It also allows for creating virtual prices for single currencies. [I have proven formulas to make predictive virtual prices for a currency in the abstract.]

My convention is to use assetid pairs for specific orderbooks (with native cryptos getting assetid == string as bits), but if it is "base":"NXT","rel":"BTC" then this would indicate that it is the weighted average value of each. The biggest problem with making good predictors is to eliminate noise and this averaging is a key factor.

It is good you already have standard timeframes, so to map your API to mine would be a matter to predefine specific "signal" to pre-selected defaults. This can be an ever expanding list and also to allow the user to define any specific one to make a new signal and have this added to the allsignals return.

due to the interface to the GUI being one signal at a time, it looks like it would just be a simplified "out" field.

OK, so what we need is createsignal API at my level, which is simply passed in the template for what to send to your library. Then during initialization, we can set the allsignals API to the list of premade signals in a config file and the user can make any custom indicator using the same createsignal API.

The GUI will use the allsignals API to make a list of signals to overlay.

Once we get the signals interfaced, it is just a matter to add the ability to issue actual orders based on the signals as inputs, with of course an arbitrary amount of NN, SVM, etc. to fine tune its risk/reward and expected ROI profile

If it is a C library, I can just link it into SuperNET

FinHive / Re: TA-JSON-RPC
« on: March 01, 2015, 03:43:01 pm »
I made InstantDEX API calls:

static char *allsignals[] = { (char *)allsignals_func, "allsignals", "V", 0 };
static char *getsignal[] = { (char *)getsignal_func, "getsignal", "V", "signal", "start", "end", "resolution", "baseid", "relid", "base", "rel", 0 };
the allsignals API currently just returns hardcoded: {"signals":[{"name":"SMA","scale":"price"},{"name":"slope","scale":"auto"}]}
just an array of ("name" "scale") with "price" meaning same as the price and "auto" meaning it needs to be scaled by the display probably in separate area from price

to get a signal you would construct a getsignal API with "signal" being then name returned from allsignals, start and end are unixtimes in seconds, resolution is defaulting to 60 but it can be set to anything, then the usual baseid/relid which would be for a specific orderbook or base/rel which would be for a higher level blend of all base/rel orderbooks

the returned JSON will have a "values" array of 5 item arrays [timestamp, OHLC], Open High Low CLose (edited)

So I think we can map the TA calls to the signals

Greetings,

I've been studying the v-corp threads on the forum and generally looking for ways to use the NXT asset exchange without violating US SEC laws (I'm a US citizen living in Australia, however all US citizens are prohibited from issuing unregistered securities).

One way that I've been considering for many months while developing my business plan was to use an offshore corporation. As a legal entity, not under US jurisdiction it could issue such assets. However that is expensive and another hassle - although something I will pursue if no other option is available. .....

the other option is to create a virtual corporation. Let me give a definition of that now:

Virtual Corporation - A semi-autonomous entity that is not directly controlled by anyone, but rather accepts proposals (contracts) and tradeable commodities (e.g. NXT) from multiple sources (preferably anonymous sources) and executes such proposals using the tradeable commodity.

Now for the concrete example/vision of how this might work - if we had the following:

   a.) a way to lock an asset issuing account by permanently destroying it's creators ability to access it and then for the account to be able to publish this fact on the blockchain - once done this account has it's puppet strings cut - it is now an entity unto itself - a virtual corporation.
   b.) a means for asset holders to be able to create propositions to vote on - essentially 'smart contracts' -- a proposition should be able to do anything a NXT account can do excepting the  manipulation of 'parent' assets. I.e. if asset owners create new assets, these are 'children' assets, and as such, owners of children assets cannot manipulate 'parent assets' as that could invite a situation akin to a hostile takeover.
   c.) a means for this sealed account to execute proposals that are activated at a specific voter approval threshold - i.e. a means to make it autonomous.
 
Anytime an asset-holder makes a proposition they are creating a smart contract that executes on the agreement of a certain percentage of asset-holders. Perhaps the contracts themselves are executed by forgers? and the proposer of the smart contract pays two fees, one to create the contract and another upon execution. Voting time-limits should be set.

The beauty here is that it can't be stifled. The government could take out the creator of the asset and even lock-up an physical business operations, but the company and it's virtual assets go on - it is autonomous. It's beyond a mere smart contract, it's a platform asset owners can use to create and execute smart contracts that separates the business managers from control of the assets themselves, hence there is nothing for a government to seize control over.

Steps:
1.) I'd like to get feedback on if this is even workable.
2.) If workable, what could be added or needs to be subtracted from the above to make it better
3.) let's get some bounties going to make this a reality - first bounty maybe should be to get a solid white paper going into detail on how this would work and how it could be implemented.
