Interesting idea. I agree having it as a single string might make cutting and pasting easier.

If we go that route, we should tweak the syntax to make it easier to parse. The syntax should indicate where the different parts are and what they mean. For example, the account numbers probably won't be 64 bits forever, so there should a terminator for them. Also, having the final part be the public key if the account is new, and a text message if the account is old, makes the syntax ambiguous. You shouldn't need access to the block-chain to understand a deposit address. (Remember also, that the account might be new when the address is created, but old when it is evaluated.)

So it should be more like:

NXT-GZYP-FMRT-FQ9K-3YQGS/Key=02ec99e42191888a4423c46b229a2a947e36d64c3a3cd8652e61c6326940a34b

where the account number ends at the slash and the "Key=" sets the public key.

NXT-GZYP-FMRT-FQ9K-3YQGS/Msg=125

would add an encrypted text message. Probably we could use shorter tags, eg "K=" and "M=", and possibly the "Msg=" could be dropped if text messages are the default, so it would become:

NXT-GZYP-FMRT-FQ9K-3YQGS/125
