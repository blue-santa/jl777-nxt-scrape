Thank you for your analysis. I did some probability calcs that show that MGW is actually significantly more secure than a centralized gateway.

1. Server redundancy is quite a benefit, especially when you consider blockchain forks. For a centralized gateway if their server (for any coin) goes on a fork, well, problems. With MGW, at least on the withdraw side everything comes to a stop. Granted response time isnt good, but I think stopping all withdraws until the fork is resolved is really the only practical solution and MGW design does this automatically. Also, MGW uses the blockchains as its database, thats pretty redundant.

2. your assumption of "if he can find an other host whatever method worked on the first one would probably work on the second" is flawed. The three MGW hosts will be proving three different sets of operational security, each based on their own proprietary security procedures. Its not like there will be a magic incantation that opens all the pathways to the MGW private keys. Now even it there was, it wont be as simple as doing a sendmoney call as all the funds are in multisig accts. Maybe this is not a big issue, but also, there wont be a single large account, but lots of small accounts. So, this magical hacker needs to crack one server host then while undetected crack another MGW host, then deal with hundreds or thousands of accounts. All without being detected as once detected, everything can be shutdown. Now how exactly the hacker will even contact an MGW server externally is a mystery to me. In any case, MGW doesnt have to run faster than the bear, it just needs to be more secure than a centralized setup. I think a social engineering hack is the weakest link, but whatever the exposure the odds are P*P vs P, where P is the probability of being cracked. Add to that the difficulty of multisig tx, add to that the lack of a centralized account. Whatever those factors are, call it X factor we have MGW being (X / P) more secure. So, if odds of one facility being cracked are .001 and X factor is 5, MGW is 5000 times more secure.

3. MGW is open. Meaning anybody can do a realtime audit on all the account balances and compare against the outstanding assets. In fact, the MGW client does this when configured with bitcoin-qt. I dont know of any centralized gateway that lets you verify they have 100% backing. Fractional reserve is always a tempting thing, but with MGW it is not possible. This is something that you overlook and I think could be one of the most powerful MGW advantages. You dont have to trust the issuer to not do fractional reserve, it is all there on the blockchain and the MGW clients can monitor this block by block. So in this most important sense, MGW is trustless. Realtime verification of unspent reserves.

MGW is thousands of times more secure than centralized gateways. MGW does NOT require trust that it has 100% reserve. MGW has lower target appeal due to no central wallet and multisig. MGW utilizes blockchain redundancy for its state data and is more immune to blockchain forks. MGW is monitored by all independent nodes running the MGW client (which is 95% the same code as MGW server) to obtain realtime verification of all MGW operations, eg. deposits and withdraws.

Now a totally blockchain solution is of course theoretically much more desirable, but I am not smart enough to figure out how to do it. The fundamental problem is "where are the private keys". If you can figure out how to store the private keys on the blockchain without allowing anybody to just empty the accounts, then let me know and I will implement it. To my knowledge, MGW is far and away the most secure gateway solution with the least trust required of any gateway in existence.

Basically, technically it is the best. The bottomline is users have to use either a centralized exchange, a centralized gateway or MGW. Well, OK, I guess people can use NXTsubatomic/NXTatomic and not have to deposit their bitcoins anywhere. Maybe for people like you that seem to want to eliminate all third party risk, that is the way to go. I do understand your point of view. So the best gateway is no gateway at all!

Thanks for the thoughtful questions. It always helps to verify my thinking when put up against intelligent probing

James

Quote from: sile16 on July 14, 2014, 03:43:33 am
Correct me if i'm wrong but the the MGW release still requires trust of the asset issuer.  From an end user point of view MGW would look identical to a single server implementation of a gateway.  (assuming the same GUI).  So, really all the MGW does is give more confidence that the server architecture is more resilient.

Benefits of MGW vs Single server Gateway assuming same GUI interface:

1.  Server redundancy in case of hardware failure, always remaining 2 servers provide ability to release funds. 
2.  Potentially slightly better security.  I say slightly because, 2 hosts would have to be compromised, but assuming a hacker can compromise 1 host, and all 3 hosts are running the same OS, and MGW software then if he can find an other host whatever method worked on the first one would probably work on the second.  As for finding the hosts, it seems like you could just crossreference all the IPs of BTC nodes as well as NXT nodes.  However, if MGW was re-implemented 3 times by 3 different programers and secured (os, firewall, etc) 3 unique ways it could achieve significantly higher security.
3.  Any other benefits of MGW that i'm missing?


Could a trustless MGW solution be built directly into NXT?  The fundamental problem with this is managing the private keys on the Bitcoin side.  There really is no way for the NXT chain to hold keys in a way for a shared pool.   However, I think what could be built is a special AE for external assets.  So, instead of having bitcoins go into a pool and then flow out, (which would require handling private keys), we do direct exchange of BTC for NXT via this special AE.  So, direct user to user Bitcoin <-> NXT transfers are done. 

1.  Special AE for BTC <-> NXT
2.  Sell orders of NXT can be placed, in the sell order, a receiving address would be generated (by the client automitically or the user could specify their own)  Would also specify a minimum amount of fees for the transaction, either as % or total amount of NXT.
3.  A buy order is placed (from an NXT wallet), (buying NXT with BTC), could potentially pay higher than minimum fee amount to get more votes.
4.  The user has to manually complete the bitcoin transaction, a QR code will provide the payment address.
5.  Some rule would specify that the BTC has to be in target address within 24-48 hours with 48+ confirmations or whatever. Biggest challenge here is transacations that don't make it into the blockchain.  If a transaction was sent out that never got confirmed in the timeframe and then for whatever reason got confirmed later that user may be out some bitcoin ... or it gets complicated.  Would have to think about the best way to handle all cases.
6.  A vote would happen by NXT nodes weighted by stake.  These nodes would be checking bitcoin balances and vote.
7.  Nodes would be encouraged to vote by paying out the couple of percentage points to a voting member based on a hash of the block with the final vote results
8.  Once the vote is completed, the next block would include the NXT transfer based on results of vote and paying to the winning vote member.

Security is based on 51% of the voting nodes to be acting honestly.  Voting nodes ideally would be DDOS resistant to reduce chances of someone altering the vote by trying to take honest nodes offline.

Note, that the BTC transfer is being traded directly from user-> user, so at no time is an NXT node in control of the Bitcoin, we only verify if the BTC transaction completes and move the NXT according based on the votes.

Partial orders and limit orders:
Partial orders could happen although each one would require a seperate vote and thus require a lot more fees to get a safe transfer.  Limit orders, having a limit order seems like it would be more problematic, since order matching is an interactive process it seems that only NXT sell orders could have a limit.  On the Buy of NXT, it would all require market orders, meaning that you put in a price it either gets confirmed immediatly or it doesn't go through at all, so a user would put in a buy order and then a couple minutes later would get the Bitcoin QR code to complete the transaction every time.

Obviously this requires changes to NXT core to achieve this but this seems much more plausible than trying to implement some method of holding bitcoin private keys. 

Other down sides are that the amount of fees may be pretty high to ensure security from the voting pool.  Another nice thing is that all forging nodes can complete their operations without any external bitcoin code.  Only the voting nodes need to have some link into the actual Bitcoin Blockchain.

Not sure how high those fees would be but if they are too high someone may just utilize a trusted gateway instead.  People could have the option of paying 5% for a decentralized voting based trade of BTC->NXT or instead use the multigateway which requires trust but has lower fees.

Say you peg at 1pNXT for 10NXT (so 1billion NXT worth of pNXT). In this case, the pNXT will operate in a similar way to the mgwBTC token. There will never be any scarcity of pNXT, and the gateway will be responsible for the fixed price of the pNXT gateway.

Now say you sell all the pNXT for 1million NXT and the peg is set at 1pNXT per 0.01NXT. Now, how do you privateSend large amounts of NXT if the total value of pNXT is only 1million NXT? Now there is huge scarcity and the value of the token will likely go up. (A lower bound of the value of pNXT is set, but will pNXT holders sell it for that much when there is scarcity and demand. Plus a higher value is needed by those needing to send large amounts.) pNXT holders could hold their pNXT for speculative purposes creating more scarcity. The system is now decentralized, but may not be operating in a healthy way. (There might be large spread on the buy/sell margins for instance)

If the pNXT is sold for 10millionNXT, then perhaps it will operate at the gateway ratio initially. But there could come a point in the future where the privacySend feature becomes more heavily used, and speculative pressures could force pNXT into a scarcity situation once again.

After having thought this all through, I'm thinking you should go back to your original plan and just hold all the pNXT tokens in the gateway account (making sure it is secured) and set the peg at 1:1 (or even make the pNXT tokens more valuable to ensure that the scarcity situation never happens). Incentivize privacy servers to hold and trade pNXT to encourage decentralization.
