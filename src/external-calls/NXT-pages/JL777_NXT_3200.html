InstantDEX will be maintaining arbitrary orderbooks in realtime, but it will not be automatically matching orders. This is why I ended up duplicating MT4 functions this weekend!

I expect that each person will look at the orderbook they are interested in and then send a offer to the one with the best price. Now, the one that receives it must decide to accept it or not. Clearly in a high volume environment that InstantDEX will become, it is not practical to do this manually. So at first I was thinking of having just a simple, "Buy/Sell X amount at ave price of Y, but not worse than Z"

Then I ran into the issue of needing some basic scripting. I found picoc and it was fantastic. 4 hours is all it took to get proof of concept execution of "hello world" script. So, once I have C interpreter in my source base, now I can add any interpreter library function that directly runs any of my NXTservices code.

What this means is that all makeoffers will be processed by the tradebot!

So you can have any Turing complete complexity level script using C combined with all the data and library functions to decide to accept or reject the offer.
Since there is no actual binding bid/ask (other than NXT AE quotes), there really is no "market order"
I will of course make standard scripts for simple things like limit orders, stop loss, etc.
But the power of all this is that you can check the orderbook status on bter and cryptsy and NXT AE all in realtime the moment a tradeoffer is there and dynamically decide to accept or reject. Now your accept/reject rate will be recorded and published so if you have many quotes but low acceptance rate, then other traders will simply not bother to makeoffers and I think the tradebots will be able to get statistics on historical acceptance rates.

Basically my tradebots will be unlike anything ever seen in the automated trading universe (as far as I know). Crypto I am newbie 6 months ago. This trading stuff I am old pro and actually quite good at it

I was thinking with my uMGW operator hat on about a e.g. python script that runs on the uMGW node, monitors the NXT and the altcoin blockchains through NXT API/RPC calls available on these nodes and stops the uMGW software in case something is wrong.

Could you outline what checks should be done?
If you see an AM containing deposit address that you did not generate, that indicates one of the other MGW operators (or someone somehow spoofing them) has gone rogue. Alerts globally, send in swat team

If you see any MGW operator putting assets for sale on AE, even unconfirmed tx, send in swat team

So if we prevent rogue operator from handing out bad addresses and also prevent making any sales via AE, then I think the only hole left is if they transfer the asset to a different unmonitored account.

So all asset transfers need to be correlated with incoming deposits. Each gateway has an ID (0, 1, 2) depends on order in the MGW.conf file. Take nxt64bits of the NXT address mod 3 to select which server is responsible for sending asset transfer.
So if any asset transfer to NXT id that isnt matching (gatewayid % 3), send in swat teams

Now, we are down to one third of transfers. The amount of the transfer must match the amount of a coin deposit and it must happen AFTER the coin deposit, so if you track all coin deposits you have the list of approved transfers. Anything else, send in swat teams

Really, this is theoretical possibility and only an issue since we dont have active realtime monitors. The reality is that any rogue operator is guaranteed to be caught and quite quickly and all for the bid side orderbook for altcoin. So, possible, but unlikely and with realtime monitoring, will be detected and stopped within a few minutes

James

Quote from: marcus03 on July 28, 2014, 02:02:45 pm
I still have my doubts...

One operator has access to 1/3 of the daily volume. Imagine we support a brand new coin with a high initial volume and we are alone (coin not yet on an other exchange). I think it would do havoc to the price of this altcoin if one operator would run with the coinassets under his control and I also think it would be quite lucrative.

Additionally users and operators have to fully trust the asset issuer, since he holds the unbound assets.

So in sum, instead of having to trust one entity (existing exchange), the user has to trust four entitities.

Since everyone else seems to be cool about it, what am I missing?
1/3 of daily volume is typically much smaller than ALL the deposits, which are unaccounted for in central exchanges. I think you are overlooking that MGW provides realtime balance info of unspent outputs and outstanding assets.
So, the block that any unauthorized activity is detected, all the MGW servers can stop processing withdraws and theoretically all the clients can start flashing alerts so people know to stop making bid orders in AE. Of course, any outstanding bids could be all filled by the rogue operator.

I would assume that in a centralized exchange, any employee could turn rogue and do some sort of similar thing. But the amount they would have access to is all the deposits.

Now you talk about "entities", I expect that uMGW operators will be just one person, so the risk exposure is 1/3 of daily balance times three people plus unbound assets in my hands. Not sure how many peoples a centralized exchange have working for them with access to funds, but any larger operation would be more than 1 person.

Due to this risk, I suggest that only trusted members are allowed to run uMGW servers. Would 10 BTC worth of altcoin assets tempt someone to throw away their reputation? Highly doubtful. More likely is that their acct gets hacked somehow, so we need to make sure all uMGW operators have high entropy passwords (somehow without requiring them to divulge it) and have adequate opsec in place

Once we get Phasing, then we can make it so 100 nodes have to agree for any asset transfer to be allowed. These 100 nodes can be a cross section of trusted members who are running MGW client that is able to monitor MGW status in realtime. Now this is not in place yet, but I cant think of any centralized solution that can lock things down like this! With Phasing, we can transfer ALL unbound assets to the Phased acct and the MGW operators wont even have any assets as all transfers would only be triggered by deposits detected by the validating nodes.

I think you are not realizing the potential of a realtime peer reviewed gateway. Now until Phasing is here I cant implement this, but my design is based on having this eliminate the few trust points in initial MGW. The coins are multisig protected, so actually it is the asset purchasers who are at risk from rogue operator and not the coin depositors. So, the people buying the assets need to be comfortable with the MGW operators and understand they are trusting these operators not to go rogue.

Also if there is some new coin that gets so much volumes, we can always spread the volumes to different uMGW clusters.

Instead of theoretical possibilities, the key question is what are the chances that the specific uMGW operators go rogue? With MGW we can personalize the risk assessment, this is also pretty unique.

James

P.S. If you think even 1,000 BTC worth of buy orders on AE would tempt me to go rogue with unbound assets, then you shouldnt use MGW.
