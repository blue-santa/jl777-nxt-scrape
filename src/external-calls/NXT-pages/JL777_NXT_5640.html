General / Re: Why isn't blockchain trimming this easy?
« on: May 07, 2014, 08:34:05 pm »
Quote from: anon136 on May 07, 2014, 07:10:14 pm
Just trim it like once per week and include the hash of the transaction table in the hashchain, everyone can keep a copy of it inorder to reference transactions and make sure they are legit, the actual blockchain with all of the transactions wouldnt actually be discarded by nodes until after the hash of the transaction table was buried under X blocks, once its buried under like 1440 blocks for example everyone will feel pretty safe that its legit.

thats just one possible solution. i dont think its that hard of a problem.
Maybe not hard, as in it is not an NP complete problem
However, from an implementation point of view it is quite messy...
NXT is now starting to get a lot of new features, each with its own set of state information. To separate out the different features would require analyzing each block and extracting anything that changes the state of a specific feature.

We now have referenced transactions, and any transaction can refer to any other. That means there is a pool of unconfirmed transactions that are not in the blockchain that most definitely affect the state. An AM tx can be conditionally linked to a poll tx.

With bitcoin it is just sendmoney that needs to be kept track of. With NXT it is all these other tx's and the type of tx's are growing every week!

I am not sure that the entropy of the entire blockchain is identical to the total state, but it does seem to be on the same order. Which means that this problem maps to lossless compressing of the blockchain. Just did a quick test on the NXT db file and got a 3:1 compression. Now, with some higher level data representations, I would estimate a 10:1 compression rate would be achievable.

Then, we can simply store compressed snapshots of the blockchain at trusted locations, publish the hash to allow new nodes to load the latest blockchain 10 times faster. The problem is with the RAM state, eg. referenced TX

Do the tx with referenced tx appear in the blockchain? Or are they purely in RAM of the nodes that were online when it was broadcast? If the latter, what happens if a referenced tx is pending for so long and a lot (of stake) of new nodes appear and then we could get a scenario where a totally valid referenced tx ends up on a fork because none of the new nodes find out about it. If nodes are exchanging unconfirmed tx, then it might open up an attack vector for injecting false pending transactions. So, either way this is done, I see potential issues...

Ok, back to the blockchain pruning. I think if we did the compression thing with one modification, it could be a nice practical tradeoff. Just parse out "expired" AM's before compressing. The expiration date can be determined by the amount of fee paid. I think a lot of the blockchain bloat will be from AM's.

James

I believe I have a solution for this that can be implemented now. In fact I already have it coded and just need to debug it.

NXTsubatomic cross chain transactions create synchronized and overlapped micropayment channels https://en.bitcoin.it/wiki/Contracts#Example_7:_Rapidly-adjusted_.28micro.29payments_to_a_pre-determined_party. This allows both parties to incrementally receive their portion in a trustless way, except for the chance that you end up short 1 micropayment, currently 1%. The percentage can be reduced, but that increases the time to complete the exchange. Currently, once the payments get going, 1% gets exchanged each second.

NXTatomic swaps implements escrow of any NXT txA for any NXT txB, eg. txX can be sending NXT or NXT assets (or any other transaction). It uses a process similar to what is described in https://nxtforum.org/nrs-releases/nrs-v0-9-9/msg10942/#msg10942
NXTatomic swap creates a linked pair of transactions, either both happen or neither happen, other than the edge case when the party that holds the trigger transaction waits until the deadline and only his tx gets through. since that is against his interest, I am ignoring this possibility.

I have a third case, which has no name yet, where I create a hybrid NXTsubatomic/NXTatomic by creating a micropayment channel equivalent for NXT. This allows direct exchange of BTC <-> NXT in a trustless way.

My concern is with https://bitcointalk.org/index.php?topic=515370.0, it isnt exactly the same, but I dont fully understand malleability and so I worry that the same sort of thing could mess up the micropayments.

My understanding is that after the funding transaction is confirmed, then receiving a signed tx using the outputs of the funding tx is as good as receiving payment. Since the funding tx went into a multisig acct that you are a signer of, you can be assured that there are no other transactions that can take precedence.

To setup a BTC <-> NXT exchange, bob wants to send nancy 1 BTC in exchange for 20,000 NXT.

bob creates an unbroadcast funding tx into a 2:2 multisig acct to get its txid. He uses that txid to create a refund tx back to himself that isnt valid for 24hrs and sends the rawtransaction to nancy. she signs it and sends it back to bob who then broadcasts the funding tx. Bob also creates any NXT tx just to create a fullhash that can be used by nancy and sends the raw data to her.

Nancy uses the NXT fullhash to create a referenced tx that pays bob 200NXT and sends it to bob. bob verifies nancy's tx decodes properly and sends nancy the signed tx bytes for .01 BTC

As each side receives the micropayment, they create a new micropayment that is 1% more, until 100% is done

Bob then broadcasts his NXT tx and nancy's 100% payment to him, they get confirmed in the following two blocks
nancy broadcasts the 100% payment to her, this invalidates the refund tx since it uses the same utxo of the funding tx

My big question is if there are any points in the above sequence where one side is more than 1% out of pocket.

However, the windows networking is all wrong. In unix, the sockets are compatible with file handles, not so for Windows. So this means I need to rewrite all the networking code of NXTsync. The original code used that method.

So I am at a crossroads. I can "simply" port the tcp and udp calls into a portable layer (which I need to write) or I can port it to libuv. Both are probably about the same amount of work. The latter runs the risk of hole punching not working as I might not be able to control the process enough to maintain the "magic", but if it works it will be a nice improvement and will ease the shift to a totally libuv based control flow. The former has a better chance of maintaining  the hole punching magic, but wont have the benefit of partial migration to libuv.
