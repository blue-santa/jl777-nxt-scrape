So, the structure of a transaction would look like this:

(D, R, HD, HA)

D - Transaction Data: timestamp, amount, sender, recipient
R - Reference (using another transaction's HD)
HD - signed hash of D

I like things when they are more concrete as I am not 100% sure what exactly you mean by all the symbols.

We already have parseTransaction which can display the details of a tx and whether it is signed or not
We will be able to create raw tx bytes and not broadcast them, thus allowing private direct exchange

Clearly, there needs to be the first tx without any dependency on any other tx. without the ability to locally generate, privately exchange and locally verify, we get impossible circular dependencies

So, A creates a payment tx to B, saves the unsigned tx bytes, signs it to get the fullhash and referencehash

Now without compromising his position, he can send B the unsigned tx bytes, the fullhash and referencehash

B can then use parseTransaction to decode the unsigned tx and most importantly obtain the hash for the unsigned tx, which is used to calculate the referencehash. Convinced that A has created the desired tx to him, B then uses the same process for a tx to A, but note that he is able to put in the referencehash of tx.A in his unsigned tx!

That is the whole key that breaks the circular dependency.

So, A can now verify the contents of B's tx to him and instruct B to broadcast tx.B (or maybe B already broadcast). Once A sees the tx.B in the blockchain, he now can trigger an atomic exchange

Now, there are a couple of scenarios. If B never broadcasts tx.B, nothing happens. if A doesnt broadcast tx.A nothing happens. What if A and/or B lied and sent garbage fullhash and the derived garbage reference hash?
Well, if A sent the garbage values, B's tx will never execute, but if B sent garbage values and A broadcasts, the he is screwed. So, it is a good thing that it is A that waits for B to actually broadcast tx.B to the blockchain.

This way, A can verify that B sent non-garbage values.

So, without math symbols that abstract stuff away, the above describes the sequence and possible scenarios.

The key is that B can create a referenced txid that hasnt been broadcast yet and know that the tx is either what A said it was or doesnt exist. sha256(sha256(hash) ^ sha256(fullhash)) makes it so that B can calculate refhash, but A can't fake it without finding another tx with the same sha256(hash), which I am assuming will be very difficult to do

James
